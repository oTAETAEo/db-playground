# Scenario 02: 지갑 잔액 차감 동시성 제어 및 락(Lock) 최적화 실험 
### 1. 비즈니스 시나리오 및 배경
   
- 도메인: 토스 실시간 간편 결제 및 송금 플랫폼 
- 상황 (유저 시나리오):
  - 사용자가 네트워크 통신 상태가 불안정한 환경에서 쇼핑몰 결제 버튼을 짧은 시간 내에 연달아 더블 클릭(Double Click) 하거나, 동일한 토스 계정에 연동된 스마트폰과 태블릿 기기에서 동시에 결제 요청을 보내는 상황입니다.
  - 매월 지정된 날짜에 여러 건의 자동 이체(정기 구독, 공과금 납부 등) 서비스들이 동일한 밀리초(ms) 단위로 한 유저의 지갑 계좌에서 동시에 출금을 시도하는 백그라운드 연산 상황입니다.
  - 여기서 네트워크 지연때문에 동시에 2번 요청이 올때 순차적으로 실행될 가능성이 있다, 그런 상황은 락으로 막히지 않는다

- 페인 포인트 (Race Condition):
  - 다수의 트랜잭션 스레드가 동시에 하나의 지갑 레코드(Wallet)에 접근할 때, 적절한 자원 보호 장치(Lock)가 없다면 스레드들이 모두 동일한 시점의 초기 잔액 데이터를 읽어가게 됩니다.
  - 이 상태에서 각 스레드가 독립적으로 잔액을 차감한 후 DB에 결과를 덮어쓰게(Write) 되면, 먼저 처리된 차감 연산 결과가 나중 트랜잭션의 덮어쓰기에 의해 증발하는 '갱신 손실(Lost Update)' 문제가 발생합니다. 
  - 이는 시스템의 자산 정합성을 파괴하여 금전적 오차를 유발하는 치명적인 금융 사고로 직결되므로, 대규모 동시 요청 속에서도 100% 데이터 무결성을 보장하는 격리성(Isolation) 제어 전략이 반드시 필요합니다.
  - 또한 네트워크의 지연으로 인해 락을 해지하고 다음 중복 요청이 오는 경우를 제어하기 위해 멱등성을 지킬 전략이 필요하다.

---

**멱등성을 지킬 전략은 reids 시나리오에서 추가로 다룬다 여기서는 동시성 문제로 인한 문제만 다룬다**
