# Scenario 02: 고빈도 백그라운드 금융 연산 환경에서의 지갑 잔액 차감 동시성 제어 및 자산 정합성 실험

### 1. 비즈니스 시나리오 및 배경

* **도메인**: 토스 실시간 간편 결제 및 정기 구독 배칭(Batching) 플랫폼
* **상황**: 매월 특정 날짜나 시간대(예: 매월 10일 새벽 2시)에 수많은 파트너사(정기 구독 서비스, 공과금 납부 시스템, 통신사 등)의 대량 배치 연산이 <br>**동일한 밀리초(ms) 단위로 한 유저의 계좌에 동시다발적인 출금(`UPDATE`) 요청을 찌르는 백그라운드 연산 상황**입니다.
* **특징**: 이 요청들은 유저의 오작동(더블 클릭)이 아니라, 각 파트너사가 유저와의 계약에 따라 청구한 '모두 정상 처리되어야 하는 독립적이고 정당한 결제 요청'입니다. <br> 따라서 외부 레이어에서 무작정 튕겨내면 안 되며, 고빈도로 밀려오는 출금 연산 속에서 **지갑 잔고와 실제 구매 완료된 총 금액의 싱크를 단 1원의 오차도 없이 맞추어야 합니다.**

---

### 2. 핵심 페인 포인트: 갱신 손실(Lost Update)로 인한 자산 싱크 파괴

만약 이 시스템에 적절한 동시성 제어 장치(Lock)가 없다면, 멀티스레드 환경에서 다음과 같은 **치명적인 자산 무결성 파괴**가 일어납니다.

```
[배치 스레드 1 (통신비)] ---- (시점 t1: 잔액 10,000원 조회) -------------------> (시점 t3: 잔액 9,900원 덮어쓰기)
[배치 스레드 2 (구독료)] --------- (시점 t2: 잔액 10,000원 조회) -------------------> (시점 t4: 잔액 9,900원 덮어쓰기)

```

1. **초기 상태**: 유저의 현재 지갑 잔액은 **10,000원**입니다.
2. **동시 조회**: 통신비 차감 스레드와 OTT 구독료 차감 스레드가 아무런 제약 없이 동시에 DB에서 유저의 지갑을 조회(`SELECT`)합니다. <br> 이때 두 스레드가 자바 메모리상에 읽어온 초기 잔액은 동일하게 **10,000원**입니다.
3. **독립 연산**: 각 스레드는 자바 애플리케이션 계층에서 독립적으로 `10,000원 - 100원 = 9,900원`이라는 차감 연산을 각자 수행합니다.
4. **싱크 파괴 (갱신 손실)**: 스레드들이 연산이 끝나는 대로 DB에 결과를 덮어쓰기(`WRITE`) 시작합니다. 1등으로 도착한 통신비 스레드가 9,900원을 반영한 직후, <br> 2등 OTT 스레드도 자신이 계산한 9,900원을 DB에 그대로 덮어씁니다.
5. **결과**: 두 파트너사는 각각 100원씩 총 200원의 대금을 정상적으로 수금했다고 판단(성공 응답 수신)하여 유저에게 서비스를 제공했지만, <br> **실제 DB 지갑 잔액은 9,800원이 아니라 9,900원이 남아있어 실제 구매 금액과 지갑 잔고의 싱크가 완전히 깨지는 금융 대참사**가 발생합니다.

---

### 3. 비즈니스 파급 효과 (Business Impact)

* **은행 자산의 손실 (Underflow/Overflow 예외 미작동)**: 실제 지갑에 남아있는 돈보다 더 많은 서비스가 유저에게 제공되어 은행이나 플랫폼 사가 그 차액을 고스란히 떠안아야 하는 자산 손실로 직결됩니다.
* **결론**: 모든 요청이 유효한 백그라운드 금융 연산 환경에서는 요청을 튕겨내지 않고 **안전하게 줄을 세우거나(비관적 락), 정확한 데이터 버전을 검증하여(낙관적 락) 순차적으로 지갑 잔액을 깎아 나가야만** 자산의 정합성을 100% 사수할 수 있습니다.

---

### 한계점 및 향후 과제
- 현실 세계에서는 네트워크 지연 등으로 인해 더블 클릭 요청이 '동시'가 아닌 '시간차를 두고 순차적'으로 인입될 가능성이 존재한다.
- 이러한 순차적 중복 요청은 DB 락 레이어를 무사히 통과하여 중복 출금을 유발하므로, DB 락만으로는 완벽한 결제 시스템을 구축할 수 없다.
- 이를 근본적으로 해결하기 위해 향후 애플리케이션 계층에서 'Idempotency Key(멱등성 키)'와 Redis를 활용하여 중복 요청을 원천 차단하는 시나리오(Redis Playground)로 확장할 필요가 있다.
