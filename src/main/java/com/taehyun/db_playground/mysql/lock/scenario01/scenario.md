# Scenario 02: 지갑 잔액 차감 동시성 제어 및 락(Lock) 실험 
### 1. 비즈니스 시나리오 및 배경
   
- 도메인: 토스 실시간 간편 결제 및 송금 플랫폼
- 상황: 사용자가 결제 버튼을 연달아 더블 클릭하거나, 여러 건의 자동 이체 연산이 동일한 밀리초(ms) 단위로 한 유저의 지갑 계좌에서 동시에 출금을 시도하는 백그라운드 동시 연산 상황입니다.
    > 예시 ) 매월 10일 새벽 2시, 수많은 파트너사(정기 구독 서비스, 공과금 납부 시스템, 통신사 등)의 배치 뱅킹 연산이 **동일한 밀리초(ms) 단위로 한 유저의 계좌에 동시다발적인 출금(`UPDATE`) 요청을 찌르는 상황.**

### 2. 핵심 페인 포인트: 갱신 손실

만약 이 시스템에 적절한 동시성 제어 장치(Lock)가 없다면, 멀티스레드 환경에서 다음과 같은 **치명적인 자산 무결성 파괴**가 일어납니다.

```
[스레드 1] ---- (시점 t1: 잔액 10,000원 조회) -----------------------> (시점 t3: 잔액 9,900원 덮어쓰기)
[스레드 2] --------- (시점 t2: 잔액 10,000원 조회) -----------------------> (시점 t4: 잔액 9,900원 덮어쓰기)
```

1. **초기 상태**: 유저의 현재 지갑 잔액은 **10,000원**입니다.
2. **동시 조회**: 100개의 요청을 처리하기 위해 생성된 100개의 스레드가 아무런 제약 없이 동시에 DB에서 유저의 지갑을 조회(`SELECT`)합니다. 이때 모든 스레드가 자바 메모리상에 읽어온 초기 잔액은 동일하게 **10,000원**입니다.
3. **독립 연산**: 각 스레드는 자바 애플리케이션 계층에서 독립적으로 `10,000원 - 100원 = 9,900원`이라는 차감 연산을 각자 수행합니다.
4. **갱신 손실 발생**: 스레드들이 연산이 끝나는 대로 DB에 결과를 덮어쓰기(`WRITE`) 시작합니다. 1등으로 도착한 스레드가 9,900원을 반영한 직후, 2등 스레드도 자신이 계산한 9,900원을 DB에 그대로 덮어씁니다. 결국 **돈은 총 200원이 깎여서 9,800원이 되어야 하지만, 앞선 스레드의 연산 결과가 증발하여 DB에는 9,900원만 남게 됩니다.**

---

### 3. 비즈니스 파급 효과 (Business Impact)

* **금융 사고 발생**: 100번의 차감 요청(총 10,000원)이 모두 정상 처리(`200 OK`)되었다고 유저에게 알림이 가고 상품도 100개가 지급되었으나, 실제 DB 잔액은 0원이 아니라 수천 원이 남아있는 **은행 자산의 싱크 불일치 대참사**가 발생합니다.
* **결론**: 대규모 트래픽 속에서도 단 1원의 오차도 허용하지 않는 금융 플랫폼의 특성상, 데이터 무결성을 100% 사수할 수 있는 **강력한 트랜잭션 격리성(Isolation) 제어 전략이 절대적으로 필요**합니다.

---

### 한계점 및 향후 과제
- 현실 세계에서는 네트워크 지연 등으로 인해 더블 클릭 요청이 '동시'가 아닌 '시간차를 두고 순차적'으로 인입될 가능성이 존재한다.
- 이러한 순차적 중복 요청은 DB 락 레이어를 무사히 통과하여 중복 출금을 유발하므로, DB 락만으로는 완벽한 결제 시스템을 구축할 수 없다.
- 이를 근본적으로 해결하기 위해 향후 애플리케이션 계층에서 'Idempotency Key(멱등성 키)'와 Redis를 활용하여 중복 요청을 원천 차단하는 시나리오(Redis Playground)로 확장할 필요가 있다.
