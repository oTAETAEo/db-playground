# Redis 분산 락을 활용한 최전방 외부 중복 호출 및 연타 방지 (Fail-Fast Idempotency) 실험 리포트

---

### 실험 조건 및 가설

#### Case 1: 동일한 멱등성 키 기반의 중복 결제(더블 클릭) 요청 방어
* **상황**: 사용자가 음식을 고른 후 결제 버튼을 연타하여 동일한 멱등성 키를 가진 결제 요청이 동시에 2번 이상 나간 상황
* **초기 조건**: 지갑 잔액 10,000원 / 음식 가격 5,000원
* **가설**: 첫 요청만 성공하여 최종 잔액은 5,000원이 되어야 하며, 이후 유입된 동일 키 요청은 레디스 인터셉터에서 비즈니스 로직 진입 전 즉시 차단(`Fail-Fast`)되어야 한다.

#### Case 2: 인프라 장애(Redis Down) 시 RDB 최후 수비 방어
* **상황**: 첫 요청 성공 후 레디스 서버가 다운되거나 키가 손실되어, 중복 요청이 1차 방파제(인터셉터)를 뚫고 비즈니스 레이어까지 유입된 상황
* **가설**: 레디스 락이 무력화되더라도 데이터베이스의 `userId + 멱등성 키` 기반 복합 유니크 제약조건(Unique Constraint)에 의해 2차 결제 생성이 원자적으로 차단되고 예외가 발생해야 한다. (이중 방어선 검증)

#### Case 3: 비즈니스 로직 실패(잔액 부족) 후 충전 및 재시도 성공 흐름
* **상황**: 첫 결제 시도 시 잔액이 부족하여 실패한 후, 사용자가 잔액을 충전하고 동일한 멱등성 키(또는 시나리오에 따른 재시도)로 다시 결제를 요청한 상황
* **가설**:
    * **1차 시도 (잔액 0원 / 가격 5,000원)**: 비즈니스 에러 발생 시 기존 레디스의 `PROCESSING` 키를 해제(삭제)하여 재시도 가능 길을 열어두고, 실패 내역은 DB 이력으로 안전하게 남긴다.
    * **2차 시도 (잔액 10,000원 / 가격 5,000원)**: 잔액 충전 후 재시도 시 레디스에 `PROCESSING` -> `SUCCESS`로 정상 상태 변경이 이루어져야 하며, 최종 남은 잔고는 5,000원이 되고 이후의 중복 요청은 완료된 결제로 간주하여 튕겨낸다.

## 알게된 점 (Lessons Learned)

토스의 공고에서 `redis`, `Zookeeper` 가 사용 기술 스택으로 있길래 찾아봄.

- `redis` 레드 락의 문제점
  - 레드락 
    - 레디스 노드가 여러 개로 분산되어 있을 때, 안전하게 분산 락(Distributed Lock)을 획득하는 공식 알고리즘 이다.
    - 예를 들어 레디스 서버 5개를 사용한다고 하면 5개의 서버에서 3개의 락을 획득해야지 락을 획득 했다고 간주 한다.
  - 문제점
    - **시계 점프 (Clock Drift) 현상**
      > 레드락은 내부적으로 "내가 락을 시도한 지 몇 밀리초(ms)가 지났지?" 라는 기계의 로컬 시간에 의존합니다. <br> 만약 5대의 레디스 서버 중 한 대의 시스템 시간이 NTP 동기화 오류나 관리자의 실수로 인해 갑자기 미래 시간으로 몇 초 점프해 버린다면, <br> 그 노드는 아직 유효 기간이 남은 락을 만료된줄 알고 스스로 지워버릴 수 있습니다. 이 틈을 타 다른 클라이언트가 락을 채가며 동시성 방어에 문제가 나타납니다.
    - **Java의 Stop-The-World (GC)와 타이밍 엇갈림** <br>
      > 클라이언트가 레디스 3대에 락을 잘 잡아서 비즈니스 로직(예: 출금)을 막 시작하려는 찰나, Java 가상 머신(JVM)의 Stop-The-World GC가 터져서 애플리케이션 스레드가 5초 동안 멈췄다고 가정해 보면 <br> 클라이언트는 가만히 있는데 레디스의 시간(TTL 2초)은 계속 흘러가게 되고 락이 풀리게 된다 이때 제3의 요청이 들어와 락을 새로 획득한다. GC가 끝난 원래 클라이언트는 락을 가지고 통과했기 때문에 하고 돈을 출금해 버립니다.  <br> **결국 두 요청이 동시에 자원에 접근하는 대참사가 발생합니다**.
      - **보완 방법** 
        - 펜싱 토큰(Fencing Token)
          - 락을 획득할 때마다 증가하는 번호(예: Lock Version 1, 2, 3...)를 발급받고, DB에 저장할 때 "내가 가진 토큰 번호가 최신 번호가 맞는지" 검증하여 GC로 늦게 깨어난 옛날 스레드의 DB 수정을 커트하는 방식
  - 결론
    - 레디스에서 사용하는 분산락 시스템은 CAP 이론에서 AP 가용성(A)과 P(분할 허용성) 시스템 이라고 할 수 있다 
    - AP 시스템이란 데이터가 아주 미세하게 유실되거나 어긋나더라도, 무조건 빠르고 가용성 높게 응답하는 것이 목표입니다.
    - 즉 금융권에서는 가용성 보다, 일관성이 무조껀 중요 하기 때문에 금융 비즈니스 로직이 있는 부분의 락은 CP 일관성(C) P(분할 허용성) 시스템인 `Zookeeper`도 같이 사용하는것 같다
