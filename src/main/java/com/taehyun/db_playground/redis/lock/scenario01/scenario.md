# Scenario 01: Redis 분산 락을 활용한 최전방 외부 중복 호출 및 연타 방지 (Fail-Fast Idempotency) 실험

### 1. 비즈니스 시나리오 및 배경

* **도메인**:
  - 토스 실시간 간편 결제 최전방 오픈 API 및 컨트롤러 레이어
* **상황**
  - 유저가 대중교통이나 지하철 등 네트워크 음영 지역에서 송금/결제 버튼을 눌렀으나, 화면 반응이 늦어지자 불안감에 결제 버튼을 연속으로 연타(Double Click)하는 상황입니다. <br> 또는 클라이언트 앱의 일시적인 렉(Lag)이나 악의적인 매크로 프로그램에 의해 **동일한 페이로드(Payload)의 결제 요청이 수 밀리초(ms) 간격으로 중복 인입**됩니다.
* **특징**
  - 첫 번째 요청을 제외한 나머지 2, 3번째 요청은 '절대 처리되어서는 안 되는 시스템 오류이자 원천 차단해야 할 어뷰징 트래픽'입니다. <br> 내부 코어 로직(DB)까지 진입시키지 않고 최전방에서 최대한 빠르게 잘라내야(Fail-Fast) 합니다.

### Redis 멱등성 방파제 처리 흐름 요약
- 1단계 (클라이언트 요청): 
  - 원래 보내려던 결제 요청 데이터(Request Body)에 고유한 멱등성 키(Idempotency-Key Header)를 원플러스원으로 묶어서 서버로 보냅니다. 
- 2단계 (스프링 시큐리티 필터): 
  - JWT 토큰 등을 검증하여 정상적인 회원인지 인증을 진행하고, 성공 시 SecurityContext에 UserId를 안전하게 탑재합니다. (이로써 악의적인 디도스성 깡통 트래픽으로부터 레디스를 보호합니다.)
- 3단계 (DispatcherServlet -> 인터셉터): 
  - 진짜 비즈니스 로직이 있는 컨트롤러를 호출하기 직전, 인터셉터가 요청을 가로채서 HTTP 헤더의 Idempotency-Key를 추출합니다. 
- 4단계 (PROCESSING 선점):
  - 레디스에 해당 키가 없다면 PROCESSING 상태를 원자적으로 저장하고 진짜 결제 트랜잭션 로직으로 진입시킵니다. 만약 이미 존재한다면 그 즉시 튕겨냅니다 (Fail-Fast). 
  만약 레디스 키가 만료된 이후에 악의적인 중복 요청이 들어오더라도, RDB의 PaymentTransaction 테이블에 걸려있는 userId + idempotencyKey 복합 유니크 제약 조건이 최후의 보루로서 데이터 오염을 원천 차단합니다.

⚠️ 실험적 제약 조건 및 구현 범위:
- 본 실험에서는 프레임워크 오버헤드를 줄이고 동시성 검증에 집중하기 위해 시큐리티는 파라미터로 모방(Mocking)하여 처리하며, <br> 3단계의 Redis 상태 조회 및 4단계의 상태 원자적 선점 및 차감 로직을 집중적으로 검증합니다.
- 레디스에 올라가 있는 `멱등성키 + UserId` 기반의 키가 만료 되어있는 상황에 들어오는 중복 요청에 대비하기 위해 RDB에 `멱등성키 + UserId` 복합 유니크를 설정 합니다.

### 2. 핵심 페인 포인트: 이중 출금(Double Spending) 및 커넥션 풀 고사

**만약 이 상황을 최전방에서 통제하지 못하고 무거운 DB 락 레이어까지 밀고 들어오게 둔다면 시스템은 양방향으로 무너집니다.**

* **비관적 락으로 막을 때의 재앙 (스레드 마비)**:
  - 유저가 버튼을 5번 연타하면, 5개의 스레드가 DB 레코드 하나를 잡기 위해 줄을 섭니다. 1번 스레드가 결제 처리를 하는 동안 나머지 4개의 스레드는 DB 커넥션을 붙잡고 멍하니 대기(Blocking)합니다. <br> 동시 접속자가 많을 경우, **이 연타 요청들 때문에 톰캣 스레드 풀과 HikariCP 커넥션 풀이 순식간에 말라붙어 시스템 전체가 먹통**이 됩니다.
* **낙관적 락으로 막을 때의 재앙 (이중 출금 금융 사고)**:
  - 버튼 연타가 거의 동시에 인입되어 1, 2번 스레드가 동일한 버전(`version=0`)의 지갑을 읽어옵니다. <br> 1번 스레드가 잔액을 깎고 커밋하여 버전을 1로 올립니다. 만약 아키텍처 설계가 허술하여 예외 핸들링이나 롤백 처리에 미세한 타이밍 이슈가 발생하거나 비동기 이벤트 큐가 꼬일 경우, <br> **유저의 돈이 이중으로 출금되거나 주문서가 중복 발급되는 치명적인 금융 사고**로 이어질 수 있습니다.
