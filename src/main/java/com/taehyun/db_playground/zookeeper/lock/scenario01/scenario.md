# [Zookeeper 도입] 외부망 연동 지연(Delay) 환경에서의 분산 락 안정성 비교 실험 시나리오

### 비즈니스 상황 설정 (PG사 도메인 빙의)

* **상황**: 가맹점(예: 배달의민족)으로부터 유저의 결제 요청이 인입됨.
* **코어 로직**: 우리 서버(PG사)는 결제 처리를 위해 **외부 카드사(신한/삼성 등)의 결제 승인 API를 호출**해야 함.
* **인프라 이슈**: 외부 카드사 내부망의 순간적인 지연으로 인해, API 응답을 받기까지 약 5초의 거대한 네트워크 블로킹(Delay)이 발생함.
* **공격적 조건**: 최초 요청이 5초간 멈춰있는(Blocking) 찰나의 순간, 유저의 새로고침 연타나 가맹점 시스템의 재시도 오류로 인해 **동일한 주문에 대한 중복 결제 요청이 동시에 인입**됨.

---

### 실험 조건 및 가설

#### **실험군 A: Redis (TTL 기반 분산 락)**

* **설정 조건**: 레디스 분산 락의 락 유지 시간(`Lease Time`)을 일반적인 실무 가이드라인인 **2초**로 설정.
* **예측 가설 및 테스트 검증 항목**:
  1. **락 유실 가설**: 외부 카드사 연동 지연이 2초를 넘어가는 순간, 레디스는 클라이언트의 작업 완료 여부와 상관없이 **TTL 만료로 인해 외부 락을 스스로 삭제**할 것이다.
  2. **동시 진입 검증 (현상 테스트)**: 1번 스레드의 비즈니스 로직이 종료되기도 전에(3초 시점), 2번 스레드가 락을 획득하여 외부 API 호출 로그를 중복해서 출력하는 **로그 오버랩 (Log Overlap)** 현상이 발생할 것이며, 이를 시계열 큐(Queue) 데이터 조회를 통해 테스트 코드로 검증한다.


* **최종 결과**: 결국 하나의 주문 ID에 대해 외부 카드사 API가 2번 호출되어 유저의 지갑에서 돈이 이중으로 출금되는 **이중 결제(데이터 오염) AssertionError**를 뱉으며 실패할 것이다.

#### **실험군 B: Zookeeper (Session 기반 분산 락)**

* **설정 조건**: Apache Curator 라이브러리의 `InterProcessMutex`를 활용하여 시간 제한 없이 **임시 노드(Ephemeral Node)** 기반으로 락을 획득.
* **예측 가설 및 테스트 검증 항목**:
  1. **세션 유지 가설**: 외부 카드사 통신 지연으로 5초 동안 톰캣 스레드가 멈춰(`Thread.sleep(5000)`) 있더라도, 톰캣과 주키퍼 서버 사이의 **네트워크 세션(Heartbeat)이 유지되는 한 주키퍼는 락을 절대로 먼저 풀지 않고 유지**할 것이다.
  2. **톰캣 스레드 풀 정체 검증 (현상 테스트)**: 1번 스레드가 5초간 멈추는 순간, 락을 얻으려는 나머지 99개의 스레드가 동시에 대기 상태(`WAITED` / `TIMED_WAITING`)로 전환되며 톰캣 스레드 자원이 순간적으로 묶이는 현상이 발생할 것이며, 이를 **ManagementFactory 스레드 덤프 API**를 활용해 테스트 코드로 직접 카운트하여 검증한다.
  3. **순차 탈출 및 방어 검증 (현상 테스트)**: 1번 스레드가 5초 후 락을 해제하면, 대기하던 99개의 후속 스레드가 순서대로 락을 얻자마자 "이미 처리된 결제"임을 인지하고 고속으로 예외 처리되어 빠져나갈 것이다. 이 **순차 출력(Fair Queueing)** 현상이 시계열 큐에 겹침 없이 적재(`Contains Exactly`)되어 있는지 테스트 코드로 검증한다.
  4. **인프라 자가 수습 검증 (현상 테스트)**: 대량의 스레드가 동시에 임시 노드를 생성하는 과정에서 미세한 병목이 생길 경우, Curator의 백오프 리트라이(`ExponentialBackoffRetry`) 가동으로 인해 **시스템의 전면 셧다운 없이 요청 처리가 자가 복구**되는지 검증한다.


* **최종 결과**: 지연 시간의 길이에 구애받지 않고 **단 1원의 오차도 없는 절대 정합성 결제 방어**에 성공하여 완벽한 초록불(`Success`)을 띄울 것이다.

---

### 검증할 최종 실험 결과 항목 (Expected Output)

* **Redis 테스트**: `AssertionError` 발생 확인 (성공 건수가 1이 아니라 2 이상으로 증가하여 동시성이 깨짐을 확인).
* **Zookeeper 테스트**: 100개의 동시 요청 중 **성공 건수는 무조건 딱 1개**이며, 총 테스트 수행 시간은 5초 이상 걸리지만 데이터 오염 없이 완벽하게 초록불(`Success`)이 뜨는 것을 확인.
